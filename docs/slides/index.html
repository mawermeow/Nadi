Section 1

Functional Requirements

使用者應該能做什麼？MVP 刻意不做什麼？

Functional Requirements

核心能力

  • 自訂 tracking items（metric / symptom）
  • 建立 daily records（數值、時間、備註）
  • 查看 summary reports 與 correlation reports
  • 探索症狀與睡眠、飲食、習慣的相對關聯

輸入型別

  • number：睡眠時數、喝水量
  • boolean：是否運動、是否喝咖啡
  • scale：頭痛程度、焦慮程度
  • text：自由描述

Functional Requirements

目前已實作

  • Items / Records CRUD 與 archived item 流程
  • Summary / Correlation report API 與 UI
  • Local-first 寫入：IndexedDB + foreground sync
  • Account linking：未登入可本機使用，登入後可雲端備份

刻意不在 MVP

  • 照片上傳、社交功能、複雜背景任務
  • 複雜 worker、microservice、external analytics pipeline
Section 2

Non-Functional Requirements

資料、安全、效能、離線與可擴展性的設計取捨

Non-Functional Requirements

Durability

  • 生活紀錄具長期價值，避免資料遺失
  • PostgreSQL 作為雲端主資料庫
  • 未來支援 CSV / JSON 匯出，降低平台鎖定

Security

  • 所有 user-owned API 必須驗證身分
  • user_id 從 session 推導，不信任 request body
  • logs 不輸出完整健康紀錄與 secrets

Non-Functional Requirements

Latency

  • 紀錄操作應低摩擦、快速完成
  • Local-first：先寫 IndexedDB，畫面立即更新
  • 複雜報表可接受較長查詢，必要時再預計算

Scalability

  • 前期個人 / 小規模 MVP，避免過度設計
  • Vercel + Neon Postgres 即可
  • 成長後靠 index、分頁、report snapshot 擴展

Non-Functional Requirements

Offline Support

  • PWA 常在手機上使用，網路可能不穩
  • IndexedDB 暫存本機資料與 pending sync operations
  • Foreground sync 在前景互動或 online 事件時執行
  • 以 version 做 optimistic concurrency，衝突需後續處理
Section 3

API Design

REST、驗證邊界、資源契約與 sync contract

API Design

Base Rules

  • REST under /v1
  • 所有 user-owned resources 需要 authenticated context
  • Date/time 使用 ISO 8601
  • 錯誤回應穩定，不洩漏 stack trace
GET/POST/PATCH/DELETE  /v1/items
GET/POST/PATCH/DELETE  /v1/records
GET                    /v1/reports/summary
GET                    /v1/reports/correlation
POST                   /v1/sync/push
POST                   /v1/sync/pull
POST                   /v1/account/device-link

API Design

Items & Records

  • Item 支援 client-generated UUID，方便 offline create
  • Update 支援 version optimistic concurrency check
  • Delete 採 soft delete + tombstones，保留歷史語意
  • Record value 必須符合 item valueType
POST /v1/records
{
  "itemId": "uuid",
  "value": 6.5,
  "recordedAt": "2026-06-17T08:00:00.000Z",
  "note": "睡得偏淺"
}

API Design

Reports

Summary

GET /v1/reports/summary

  • metrics：avg / min / max / total / count
  • symptoms：occurrence count / avg severity

Correlation

GET /v1/reports/correlation

  • window-based 觀察 symptom 前後的 metrics
  • 回傳 correlation score 與 sample size
  • 協助找出 symptom 與其他紀錄之間的可能關聯

API Design

Sync Contract

POST /v1/sync/push
{
  "deviceId": "...",
  "operations": [{
    "operationId": "...",
    "entityType": "record",
    "operationType": "create",
    "entityId": "...",
    "baseVersion": 1,
    "payload": { ... }
  }]
}

POST /v1/sync/pull
{
  "deviceId": "...",
  "checkpoint": { "until": "...", "cursor": "..." }
}

Pull 使用 checkpoint + cursor 分頁，降低多裝置大量變更時的遺漏風險

Section 4

High-Level Design

Runtime 架構、元件分工與資料模型

High-Level Design

Architecture

Client PWA
  │
  ├─ IndexedDB (local-first UI)
  │
  │ HTTPS
  ▼
Next.js on Vercel
  │
  ├─ Route Handlers / Server Actions
  ├─ Auth middleware (Better Auth)
  ├─ Validation + Service layer
  │
  ▼
Neon PostgreSQL (Drizzle ORM)

前期刻意不引入：Redis、Message Queue、Object Storage、Microservices

High-Level Design

Runtime Flow（Phase F）

  1. UI 優先讀寫 IndexedDB
  2. Local write 建立 pending sync operation
  3. Foreground sync push pending / failed operations
  4. Server 以 version 檢查衝突
  5. Client pull remote changes + tombstones
  6. Merge 回 IndexedDB，UI 重新讀 local data

High-Level Design

Database Model

users
items        (user-defined tracking definitions)
records      (time-series observations)
report_snapshots (optional, for heavy reports)
sync_device_sessions (per-device checkpoint state)
  • 所有 user-owned tables 以 user_id 隔離
  • Item archived，不 hard delete，保留歷史語意
  • Record recorded_at 以 UTC 儲存，UI 再轉時區

High-Level Design

Index Strategy

items(user_id)
items(user_id, type)
items(user_id, archived)

records(user_id, recorded_at DESC)
records(user_id, item_id, recorded_at DESC)
records(item_id, recorded_at DESC)

對應常見查詢：全部紀錄、單一 item 趨勢、報表時間區間聚合

Section 5

Deep Dive

關鍵設計判斷、取捨與 edge cases

Deep Dive

為什麼用 items + records？

系統核心不是固定欄位，而是讓使用者自己定義觀察項目。

不建議

daily_records {
  sleep_hours,
  coffee_count,
  headache_level
}

採用

items  → 定義項目
records → 儲存觀察

EAV-like，但不過度泛化

Deep Dive

為什麼選 PostgreSQL？

  • 時間區間查詢、group by、join items/records 很自然
  • 適合 summary 與 window-based correlation
  • 資料一致性與匯出能力較完整

不優先 DynamoDB / Firebase：探索式分析與 ad-hoc query 成本較高 

核心判斷：
這不是待辦 App，而是個人時間序列資料分析系統

Deep Dive

Correlation 怎麼做？

  1. 找出 symptom 發生時間點
  2. 回看 symptom 前 N 小時內的 metric records
  3. 計算 correlation score 與 sample size
  4. 回傳候選項目與描述，供使用者自行回顧
"description":
"在目前資料中，頭痛發生前 24 小時內，
睡眠時間偏低時較常出現頭痛。"

Deep Dive

Account & Device Linking

  • 未登入：anonymous local-first，只寫本機
  • 登入後：需明確 device-link，避免隱性資料覆蓋
  • Sync identity 以 authenticated user 為準
  • 保留「先本機使用、之後再備份」的低門檻體驗

Deep Dive

Edge Cases

  • 刪除 item → archive，不 hard delete
  • 修改 unit → 歷史資料語意可能改變，前期保守處理
  • value 型別錯誤 → API validation + 400
  • 離線重複建立 → client_generated_id 去重
  • 樣本數不足 → 降低 score 可信度或提示資料仍不足

Deep Dive

效能演進路線

  1. 前期：直接查 records 即時計算
  2. 中期：強化 index、限制 report date range
  3. 後期：report_snapshots 預計算 + Cron / worker

原則：不要為尚未發生的效能問題過早引入複雜基礎設施

Closing

Final Recommendation

Next.js PWA + Vercel + Neon Postgres + Drizzle + Better Auth

第一版重點：
1. 快速建立自訂項目
2. 低摩擦紀錄 daily signals
3. 探索 symptom 與其他紀錄之間的可能關聯
